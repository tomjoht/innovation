Disruptive tech: Static site generators

Lightweight, flexible, no database

- Tech works using command line.

- Fits well into the API doc space.

- 50 options but most work similarly

- Author in Markdown, which is glorious.

Static site generator characteristics

  • Lightweight. No database — output is just static files.
  • Fast. As static, pre-built files, pages load < 1 second.
  • Usability. Write in Markdown. Add HTML or JS as needed.
  • Flexibility. Complete control to define entire site.
  • Secure. As static files, very secure.
  • Suitable for hackers. Treat content just like code.

Jekyll is most popular

- Preview mode to see changes in browser.

- Uses Liquid templating language for advanced logic.

- Used in major sites: getbootstrap.com, healthcare.gov, more.

Live preview rebuilds with each save

- Live preview allows you to see

- Build time is under 2 seconds

- No need for MD editor w/ split view

- DITA not only tech for cond. filtering

Documentation theme I built on Jekyll

- Used Bootstrap framework and jQuery plugins.

- Vanilla theme starts simple, then grow complexity.

- Liquid allows you to do all DITA functionality + more.

5 reasons why static site generators can be a disruptive technology

Reason 1: Snug fit into web technologies

HTML, JS, CSS

- Can't rely on UX. Tooling accessible even to non-UX.

- These three are basics of everything on web.

- So many resources online to learn.

Bootstrap

- The bootstrap grid -- makes responsive design easy.

- Whole templates built on Bootstrap.

- Add new components like magic with styles

- Lots of JS Fiddles showing how-tos, e.g., collapsible section.

jQuery

- jQuery plugins for everything -- selecting, filtering, animating.

- Know enough jQuery to integrate & world expands.

- In my Jekyll theme, jQuery for sidebar, drop-downs, mini-TOCs, expand/collapse.

Markdown

- A ton of web dev tools process Markdown.

- Simple way to express HTML.

- No tags to get in the way, so it's very readable even in code editor.

- Where you fall short, use HTML, just be careful of spacing.

UX talent will help when platform is web

- Experience in working w/ Alan. His tools: Esprima, Gulp, Parse.js., Estraverse, Grunt.

- Response to using XML? Make easy to extract content.

- Extensive doc portal solutions require UX skillset.

Reason #2: Scale infinitely

Grow as needed without budget constraints

  • Allow unlimited number of authors.
  • No price per "seat."
  • Avoid free-for-all content junkyard of wikis.
  • Handle any type of content.

- Main criticisms: Static site gens work only for small projects.

- Acknowledgment: Could be true. But consider scalability of Wikipedia.

- No need to create fancy specializations for custom content types.

Allows everyone to be a tech writer

One person writing from one perspective in one company, especially an outsider to the actual business context of the user, can't possibly cover all the information needs for every user.

Reason #3: Content APIs

Growth in REST APIs

- Websites are more than standalone pieces of content.

- Everything is a mashup now.

- Avg web developer works with 8-10 APIs per site.

- Web is expanding into a true intertangled, interwoven web.

-

Healthcare.gov site as API

Why not create help content APIs?

- Push content into JSON format.

- Allow apps to pull in where needed.

- Easy way to control in-app help.

- Put help in exact spot where needed.

Populate UI text dynamically from API

- Devs just code IDs in places.

- Must enable CORS on your domain.

- Counterargument: Lacks scope of a web search w/ many results.

- Rebuttal: In ideal scenarios, avoid search scenarios altogether.

Reason #4: Push content into templates

Custom templates for custom content types

- Single sourcing paradigm already familiar.

- Take it one step further: define the formats/structures you want.

- Push to JSON, CSV, API doc, other.

Iterate through items


{
  "entries": 
[
    {% for page in site.tooltips %}
    {
      "id"    : "{{ page.id }}",
      "body": "{{ page.content | strip_newlines | replace: '\', '\\\\' | 
               replace: '"', '\\"' }}"
    } {% unless forloop.last %},{% endunless %}
  {% endfor %}
]
}

- for loop iterates through content.

- With each item in set, puts id & body here.

- This is Liquid templating language.

- Do similar tactic for CSV and Salesforce.
